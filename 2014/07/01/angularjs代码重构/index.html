AngularJS代码重构

翻译：Get Your Hands Dirty Refactoring in AngularJS
精华总结：

  1. 没有和scope有关系的代码，都可以移到service里面去

    一些常见的问题

  • view很难debug，不知道scope的变量具体来自哪里
  • 有些变量你会以为是window的对象
  • 有的时候你想从parent scope去访问一些child scope的成员，这个时候你可能会用到了$broadcast/$on去实现这个需求。或者为了方便直接把这个child scope移到parent scope
  • 对用户的一些操作，不知道具体发生了什么，因为有好多的事件同时执行了
  • 至少有一个service里面超过了1000行的代码，好多代码都是一些随便的设置
  • 在使用$http的时候，用的稀稀拉拉的，没有固定的地方去安排这些代码

上面说的这些都是一些错误，而且这些错误很容易发生。
为了解决这些错误，我们要做的就是：重构代码！

为什么要重构

在花钱让你写代码的人看来，重构代码是很不值得的，因为重构的过程需要耗费明显的时间，但是时间花了却没有什么明显的变化。
而对于程序员来说，会导致，重构后，修改了一部分bug，却又带了一些其他的bug。很少有人会因为重构，却引入了一些可以炫耀的新功能。
那么为什么还要重构。
重构是一个预付费的事情，可能短期看不出效果，但是长远来说还是很有帮助的。
如果你现在重构，在以后，经过重构的代码将会耗费比较少的时间去完成一些新功能的开发。
接下来，让我们看看一些常见的问题和解决方案！

复杂的controller间的继承关系

你有很多的嵌套的controller，并且存在过度依赖的scope继承关系。在这种情况下，模式是很有帮助的。我可以提供一些指南。
html代码:

<div ng-controller="A">
    <div ng-controller="B">

    </div>
</div>

controller A:

myModule.controller('A', function($scope) {
  $scope.foo = 'bar';
});

controller B:

myModule.controller('B', function($scope) {
  // ...things...
});

对于上面的代码，我们应该把foo移到controller B里面，因为在controller A里面根本没有使用foo
controller是用来提供逻辑处理和数据给view的。如果一个controller的view没有用到的成员，这些成员都不应该存在于这个controller里面。

从parent controller里面去提取出一些功能可能比较困难，比如
html:

<div ng-controller="A">

    <div ng-controller="B">

    </div>
</div>

js:

myModule.controller('A', function($scope) {
  var f = 'foo',
    b = 'bar';

  $scope.foo = function foo() {
    return f;
  };

  $scope.bar = function bar() {
    return b;
  };

  $scope.baz = function baz() {
    return f + b;
  };
});

myModule.controller('B', function($scope) {
  // ...stuff &amp; things...
});

正如上面讨论的那样，我们应该要把foo()放在controller B里面，变量f也不应该放到 B里面。
但是如果我们把f 变量移走了，baz() 方法就会报引用的错误。
那我们该怎么办呢？
很简单，使用service或者factory。
修改过后的代码：

myModule.service('Placeholder', function() {
  var f = 'foo',
    b = 'bar';

  this.foo = function foo() {
    return f;
  };

  this.bar = function bar() {
    return b;
  };

  this.baz = function baz() {
    return f + b;
  };
});

myModule.controller('A', function($scope, Placeholder) {
  $scope.baz = Placeholder.baz;
  $scope.bar = Placeholder.bar;
});

myModule.controller('B', function($scope, Placeholder) {
  $scope.foo = Placeholder.foo;
});

这样修改后，提高了代码的可测试性，而且让scope变的也比较干净。
我发现，如果合适的时候service，代码会组织的很好，而且app也很容易被测试。
所以当有些功能和scope是没有关系的时候，我们需要考虑把这些方法移到service里面。
译者注：注意，没有和scope关联的方法，放到service是很不错的实践方法。

如果你有一个function在controller的scope里面，但是没有和scope本身有联系，这里的联系指的比如和scope的变量。

  • 你想要让function可测试的
  • 建一个一次性的service、factory给这个方法，好像比较愚蠢
  • 这个function仍然没有属于一个scope

你的选择是很有限的，我发现的解决方法可能不是很好，但是可以用：

var MyCtrl = function MyCtrl($scope) {
  $scope.foo = 2;

  $scope.bar = function bar() {
    return MyCtrl.baz($scope.foo) + $scope.foo;
  };
};

MyCtrl.baz = function baz(foo) {
  return foo + 2;  
};

myModule.controller('MyCtrl', MyCtrl);
